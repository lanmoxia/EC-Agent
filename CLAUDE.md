## 角色边界（最高优先级）

**Claude Code 在本项目的角色是"视频分析 + 提示词输出"**：分析视频 → 输出结构化报告 → 输出对应平台提示词。

- ✅ 收到视频 → 分析 → 输出结构化报告 → 输出豆包/可灵提示词

- ❌ 不主动做自动化架构设计
- ❌ 不主动重写 conversation log 里的旧提示词

## 视频分析能力

默认走 video.js 视频直传模式（qwen3.5-omni-plus 原生多模态：能听音频、看口型、感知节奏）。视频过大时自动降级。

```sh
node video.js "<视频路径>"                         # 默认视频直传（推荐）
node video.js --frames [N] "<视频路径>"             # 降级抽帧（丢音频/节奏）
node video.js --url "<视频URL>"                     # 公网 URL（基本不用）
```

可选参数：`--keep` 保留临时帧目录用于调试。

## 视频分析输出规范（9 段）

每次完整分析按以下顺序输出，不偷懒不跳段，每段尽可能详细：

**报告开头（第一行）**：先输出声部摘要，例：`声部：画外女问 → 画内女答` 或 `声部：单人口播`，再接正文9段。

1. **视频基础信息** — 总时长（精确到0.1s）、画幅比例、拍摄类型、镜头数、是否一镜到底、整体拍摄质感
2. **第一帧精确构图** — 视频第0秒的静态画面布局（最重要段落，必须写全）：路/地面2D动线方向（上下+左右同时写）、各层元素的画面象限位置、角色位置和占画比、角色朝向、镜头起始机位、**角色第0秒初始体态（走路中/静止站立/说话状态，有无微侧身等）**
3. **角色外观档案** — 每个人物单独一条：视觉年龄（具体数字）、**面容气质（五官风格/日常感/气质描述，如"面容姣好自然亲切"，不写模特感）**、发型发色、服装逐件（颜色+款式+材质感）、下装、鞋、包/配饰、皮肤色调、**体型身高比例**、**典型站姿/走路体态**；多人额外写相对位置和互动方式
4. **光线与色调** — 光源类型+方向+质感；色温具体描述（不只写"偏暖"）；饱和度+具体感受；主色块及其画面位置；整体视觉风格；天气/时段线索（室外必填）；背景虚实程度
5. **时间轴台词** — 格式：`M:SS.s-M:SS.s | 说话人 | 画内/外 | "台词，停顿用//，换气用/"（语气 语速）`；语气只用低强度词；开头结尾各3秒务必仔细听
6. **动作与道具时序** — 格式：`时间点 | 动作描述 | 道具状态`；道具每次状态变化单独记录（持握方式+画面位置+封面朝向+倾斜角度）；步态速度；手势细节（哪只手、手指状态）
7. **镜头运动时序** — 格式：`时间段 | 镜头类型 | 运动向量（方向+速度感+起止角度） | 景别`；手持晃动程度
8. **场景与背景细节** — 场景类型；前/中/远景各有什么（含象限位置）；地面材质；固定道具（护栏/绿植/建筑等）位置；动态背景元素；空间感
9. **复刻风险提示** — 双人同框难点、手部/持书动作难度、台词密度风险、镜头运动难度、时长是否超豆包10s上限

**错误复核（静默执行，有问题才输出）**：每次分析完都执行以下检查，无问题则不输出这一段；发现问题时只列修正点：

- 开头/结尾3秒边界漏听核查
- 屏幕字幕 vs 真实音频一致性
- 台词是否被拆错/截断/漏掉/归错说话人

### 必须输出 vs 不要输出

- ✅ 第一帧精确构图 / 角色逐件服装 / 视觉年龄 / 光源方向 / 主色块位置 / 台词停顿标注 / 道具逐次状态 / 镜头运动向量 / 背景各层元素
- ❌ 道具书封面配色/文字/图案 / 屏幕字幕/贴纸/促销文案 / 装饰特效 / 背景音乐 / 强情绪词 / 主观氛围评价 / 影视化修辞

## 批处理默认策略（少打断用户）

**核心原则：用户发完消息就可以去忙别的，回来看到的是完成的报告，不是等待确认的问题。视频分析全流程（分析→报告落盘→校验→输出提示词）不在中间停下来确认，批量任务同样不停。遇到不确定的情况，自行做决策，结果里注明即可。**

以下决策全部**不问直接做**：

| 场景 | 默认动作 |
| --- | --- |
| 视频 ≤ 14MB | `node video.js` 直传 |
| 视频 > 14MB | ffmpeg 压缩 `-c:v libx264 -crf 28 -preset fast -c:a aac -b:a 96k`→ `_small.mp4` |
| 压缩后仍 > 14MB | 再压 crf 32，仍超 → `--frames` 降级，报告头部注明 |
| omni 报"看不清/听不到" | crf 23 重压重跑 |
| 背景音乐明显盖声 | ffmpeg 音频增强（见下）→ 重跑 |
| 视频模式失败 | 自动降级 `--frames`，注明精度有限 |
| 复核 pass | **每次必跑**，不问 |
| 时间轴校验 | 报告落盘后立即跑 validate.js，**每次必跑** |
| 多视频批处理 | TaskCreate 建任务列表 → 串行跑 → 汇总 |
| 公网上传 | **默认不上传**，直接 inline base64（境外 URL 百炼下载超时） |

### 报告落盘规则

- 目录：`E:/claude-vision-skill/reports/`（不存在就 mkdir）
- 命名：`<视频名去扩展名>_<YYYYMMDD>.md`
- 多视频额外生成 `_summary_<YYYYMMDD>.md`
- 会话里只贴路径 + 2-3 行摘要，完整内容写文件

### 时间轴校验（每次必跑）

报告落盘后立即运行：

```sh
node validate.js "<视频路径>" "<报告路径>"
```

- ❌ 错误：修正报告中对应段落后重新落盘
- ⚠️  警告：对可疑时间段单独追问模型确认，结果注明"人工确认"
- ✅ 通过：无需操作

### 音轨增强（背景音乐盖声时）

模型报"听不到"或台词明显识别错误，先做音频增强再重跑：

```sh
ffmpeg -i "<原视频>" -af "afftdn=nf=-20,highpass=f=150,lowpass=f=4000" "<原视频名>_voice.mp4"
node video.js "<原视频名>_voice.mp4"
```

增强后仍失败 → 降级 `--frames`，报告头部注明精度有限。

### 5 类必停清单（只有这些才打断用户）

1. 视频文件不存在/路径错误
2. 压缩后 omni 仍报"看不清关键文字"或"听不到音频"
3. 批 5 个挂 2 个以上
4. 用户主动说"暂停/停/等等"

必停时给"1 句话状态 + 推荐方案"，不空泛地问。

## 代码架构

### CLI 脚本（项目根目录）

**vision.js** — 图片分析。将本地文件 base64 编码或直接传 URL，以 `image_url` + `text` 格式发送到 DashScope。

**video.js** — video.js 是 vision.js 的超集，支持三种模式：

| 模式 | 触发方式 | 原理 |
| --- | --- | --- |
| 视频直传（默认） | `node video.js <file>` | base64 编码整个视频，`video_url` + `text` 发送，`max_tokens: 4000` |
| 公网 URL | `--url <url>` | 直接传 URL 字符串，无大小限制 |
| 抽帧降级 | `--frames [N]` | ffprobe 获取时长 → ffmpeg 抽取均匀帧 → 多图发送 |

**API 格式：** 两个脚本均使用 OpenAI 兼容格式（`chat/completions`），`messages[].content` 为数组，每个元素类型为 `image_url`、`video_url` 或 `text`。更换 `DASHSCOPE_BASE_URL` 和模型名即可接入其他兼容 provider。

**大小限制：** video.js 内联 base64 硬限 50MB（超出报错）。批处理策略里的 14MB 阈值是 Claude Code 自动压缩的触发点，不是 API 限制。

### server/ — Express Web 服务

与根目录 CLI 脚本平行的完整 Web 后端，将视频分析能力封装为 REST API + SSE 进度推送。

**开发命令（在 `server/` 目录下执行）：**

```sh
npm install          # 首次安装依赖（需要 node-gyp / better-sqlite3 编译）
npm start            # 生产启动 (node app.js)
npm run dev          # 开发启动（nodemon 自动重载）
npm run lint         # ESLint 检查 (eslint src/**/*.js app.js)
```

**数据流：** `POST /api/tasks` → `queue.service` → `analysis.service` → `video-analyzer.js` → `dashscope.js` → Qwen API，进度通过 `GET /api/sse/:taskId`（EventSource）实时推送。

**REST API 端点：**

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/api/tasks` | 上传视频（+可选参考图 `refImages` + 台词 `scriptMode/scriptJson`），创建分析任务 |
| GET | `/api/tasks` | 任务列表（分页：`?page=1&limit=20`） |
| GET | `/api/tasks/:id` | 单个任务详情 |
| POST | `/api/tasks/:id/compare` | 上传生成视频，创建对比分析任务（`taskType: "comparison"`, `parentTaskId`） |
| GET | `/api/tasks/:id/comparison` | 获取该任务最新的对比分析任务 |
| GET | `/api/sse/:taskId` | SSE 进度推送（EventSource 订阅） |
| GET/POST | `/api/reports` | 报告列表 / 创建 |
| GET/POST | `/api/accounts` | 账号管理（抖音账号） |

**SQLite 数据库** 存放在项目根目录 `claude-vision.db`（相对 `server/` 为 `../../claude-vision.db`），WAL 模式，4 张表：`tasks`、`reports`、`feedback`、`accounts`。迁移在 `src/models/db.js` 启动时幂等执行，无需手动跑迁移命令。

**queue.service.js 并发控制：** 内存队列，并发上限由 `QUEUE_CONCURRENCY`（默认 3）控制，不依赖 Redis 等外部组件。

**video-analyzer.js** 是 `video.js` 的服务端移植版，增加了：自动压缩降级（crf 28→32→抽帧）、参考图注入、用户台词注入，逻辑与 CLI 版保持同步。`src/lib/validator.js` 是 `validate.js` 的库版本（供 `analysis.service` 调用，无 CLI 包装）。

### client/ — Vue 3 前端

**开发命令（在 `client/` 目录下执行）：**

```sh
npm install       # 首次安装
npm run dev       # 开发服务器（port 5173，/api 代理到 localhost:3000）
npm run build     # 构建到 client/dist/（server 生产模式从此处服务静态文件）
npm run lint      # ESLint
```

**技术栈：** Vue 3 (Composition API) + Vite + Tailwind CSS（shadcn/ui 风格设计系统，CSS 变量 token）+ Pinia + Vue Router + Axios + lucide-vue-next 图标

**路由：**

- `/` → HomeView（上传视频 + 最近任务快捷入口）
- `/tasks/:id` → TaskView（分析进度 SSE 实时日志 + 完成后展示报告）
- `/history` → HistoryView（任务列表分页 + 导出反馈 JSON）

**核心组件：**

- `VideoUploader.vue` — 拖拽上传 + 参考图（最多5张，带标签分类）+ 台词（单人/多人/画外音三模式）
- `ReportViewer.vue` — 四 Tab（AI报告 / 人看摘要 / 豆包提示词 / 时间轴校验）+ 侧挂书签反馈按钮

- `CompareViewer.vue` — 展示对比结果：系统差异分析 + 提示词重新优化
- `TaskCard.vue` — 任务列表卡片（点击跳转详情）

**设计系统：** `src/assets/index.css` 定义 CSS 变量（light/dark 两套），Tailwind 配置引用这些变量。所有 UI 组件在 `src/components/ui/`，使用 `class-variance-authority` 管理变体。

**API 层：** `src/api/axios.js` 统一封装（响应拦截器自动取 `data` 字段），`tasks.api.js` / `reports.api.js` 分别封装各端点。

## 提示词撰写规范

视频分析报告完成后，紧接着输出提示词。平台由用户指定，未指定时按时长自动判断（≤10s → 豆包，>10s → 可灵）。

### 通用规则（豆包 + 可灵都适用）

**核心原则：像跟朋友描述你脑子里想象的画面，不是给AI写说明书。** 越白话越自然效果越好——豆包等中国短视频模型训练数据以自然语言为主，日常叙述比指令格式效果更好。

**格式**：叙事散文，不用【中括号】结构。所有约束（镜头/字幕/画幅/真实感）全部融入自然叙述里，不单独列技术头。

**参考图原则**：

- ✅ 给：干净场景图（无角色）+ 道具书图
- ❌ 不给：原视频截帧（含角色动作的帧）——角色截帧会让模型"学"原视频的外观，盖过文字描述，导致服装/镜头行为照抄原版

**真实感写法**（融入场景描述，不单独一行）：

- 室内：`像手机随手拍的` 或 `真实普通住宅`
- 户外：`手机随手拍质感，画面自然不做作`

**反表演约束**（融入动作描述）：

- `动作自然，不要摆拍感`
- `语气像和邻居/朋友聊天，不要主播腔`

**★ 人味写法：情境→人的反应→镜头跟着走**（核心，优先于技术描述）：

提示词写的是一个真实场景里自然发生的事，不是演员按剧本做动作。先写是什么情境让这一切发生，再写人怎么反应，镜头跟着人走。

- ✅ `被画外人叫住，女子侧身回头说…`（有原因，动作自然）
- ❌ `角色向镜头方向转身`（无原因，像在表演）

常用人味触发词：被叫住、被喊住、侧身回头、停下来、扭头看、自然走入

**镜头词汇表**（只用这些，够用，不造新词）：

| 类型 | 写法 |
| --- | --- |
| 基础质感 | 手持手机（微晃） |
| 视角（重要）| 第一人称视角——与"手持手机"叠加，画面变成路人拿手机遇到这个人的感觉，不是拍摄者在拍表演，真实感显著提升 |
| 固定 | 固定镜头 / 镜头跟随固定��景 |
| 机位角度 | 轻微俯视 / 轻微仰视 |
| 运镜 | 前推 / 回拉 / 左侧横移 / 右侧横移 / 镜头跟随角色 |

**风格锚点**（场景有明确风格时加一句）：`邻居种草推荐风格` / `朋友圈记录感` / `街头随手拍感`

**色调还原**（每条提示词必加，从分析报告第2段色调字段直接转写）：

| 分析结论 | 写入提示词 |
| --- | --- |
| 高饱和度 + 偏暖 | `画面色调偏暖，颜色鲜艳` |
| 中等饱和度 + 偏暖 | `画面色调温暖自然` |
| 户外强自然光 | 加：`阳光充足，画面明亮` |

**色块精准写法**（有明显主色时加一句）：`橙色上衣明亮但不荧光，绿色背景自然，整体不要灰暗。`

**道具书写法**（每次涉书必须包含以下全部，不省）：

| 要写 | 示例 |
| --- | --- |
| 持握动作 | 从车筐里自然拿起 / 举起 / 指向 / 晃一晃 |
| 空间姿态 | 拿到胸前偏下位置 / 伸直手臂平举 / 放在桌上 |
| 封面朝向 | 封面大致朝向镜头 / 书脊朝镜头 / 反面朝镜头 |
| 强制约束 | **禁止翻开书** |
| 书的外观 | **不描述**（颜色/文字/图案由参考图覆盖） |

**台词格式**：

```text
[角色]自然说："[用户提供的新台词]"
画外[女/男]声说："[台词]"
```

**配角处理**：只写位置约束 + 被动动作，不给配角戏份。
示例：`孩子坐在后座安静陪衬，可以看妈妈或看书，不说话。`

**台词顺序**：有画外音时，画外音台词必须写在前面，画内台词写在后面。

**角色年龄推断规则**（提示词生成时必须遵守）：

| 台词/场景线索 | 视觉年龄写法 |
| --- | --- |
| 三年级 | 约8-9岁，身形偏小，面孔稚气 |
| 四年级 | 约9-10岁 |
| 五年级 | 约10-11岁，小学高段 |
| 六年级 | 约11-12岁，少年感 |
| 高一 | 约15-16岁，青少年 |
| 孩子是小学低段（1-3年级） | 妈妈/家长视觉年龄约30-35岁 |
| 孩子是小学高段（4-6年级） | 妈妈/家长视觉年龄约33-38岁 |

面容写法：用"面容姣好""气质自然亲切""面容清秀干净"等日常词，不写"漂亮/精致/模特感"，贴近普通人真实气质但比平均好一点。

**豆包提示词核心链路（按此顺序写，不跳段）**：

镜头怎么拍 → 谁在什么地方 → 从哪往哪走 → 画外音触发 → 角色立刻反应 → 稳定到什么景别 → 角色说完整台词 → 书参考图 → 手机质感和光线

**豆包提示词基础模板**：

```text
手持手机镜头跟拍，【人物描述】在【场景】里，从画面【方向】向【方向】走，【画外音/第一句台词】。人物听到后【反应动作】，镜头变成【中近景固定/近景固定】，人物说：【完整台词】。【书本道具说明】。手机随手拍质感，光线明亮柔和，画面清楚，不加字幕。
```

填写说明：

- 【人物描述】：视觉年龄数字 + 面容一句话 + 单件上衣颜色款式 + 下装（不写叠穿）
- 【场景】：一句口语化场景（"小区快到单元门的道路上"）
- 【方向】：左/右，同时满足2D空间逻辑
- 【画外音/第一句台词】：画外音写`画外音：xxx`，直接开口写台词
- 【反应动作】：口语化描述（"立马扭头看向镜头"），不写技术动作
- 【景别】：`中近景固定` 或 `近景固定`，融入句子不单独列
- 【完整台词】：原始台词完整写出，不截断
- 【书本道具说明】：`书参考参考图`（有参考图时），或`双手把书举到胸前，封面朝向镜头`

> ⚠️ 旧的3版策略（极简直白/空间构图/全要素精简）已实测废弃，不再使用。

### 豆包版（≤10s，单条提示词）

**触发条件**：视频总时长 ≤10s，或用户明确指定。

**参考图引用**：用数字序号，用户按提示词顺序自己排图。

- 书：`书参考用户提供的参考图`
- 角色（有参考图时）：`爸爸参考第N张图片，女儿参考第N张图片`

**角色年龄感**（无参考图时必写，有参考图时省略）：

从台词中识别年级，映射到视觉年龄写进角色描述：

| 年级 | 视觉年龄 | 描述要点 |
| --- | --- | --- |
| 三年级 | 8-9岁 | 身形偏小，面孔稚气，小学低段感 |
| 四年级 | 9-10岁 | 略高于三年级，仍有稚气 |
| 五年级 | 10-11岁 | 小学高段，开始有少年感 |
| 六年级 | 11-12岁 | 明显少年感，体型接近初中生 |
| 高一 | 15-16岁 | 青少年外貌，与小学生有明显视觉差距 |

**Seedance 硬限制**（实测无论怎么写提示词都控制不了，不要浪费字数描述）：

| 无法控制 | 说明 |
| --- | --- |
| 左/右手持书 | 随机，靠可灵+参考图解决 |
| 书封面文字/图案 | 模型自己编，靠参考图解决 |
| 多层叠穿服装 | 必然合并成一件，写单件 |
| 具体场景道具 | 护栏/电动车型号等无法精确，靠可灵 |

**服装写法**（豆包专用）：角色服装只写**单件**，例：`亮橙色上衣，深色裤子`。不写叠穿（开衫+内搭），否则豆包会随机合并颜色。

**双工具分工**：豆包只负责验证动作框架（一镜到底 / 语气 / 基础构图是否对）。精准复刻（手持哪只手 / 书封面 / 具体场景道具）改用可灵+参考图。

**开场构图**（豆包没有参考图兜底，必须在提示词里精确描述第一帧）：

从分析报告第2段提取以下信息，写成开场空间描述：

**核心原则：AI以2D画面坐标理解空间——"远处"="画面上方"，必须同时写上下+左右方向。只写左右会导致空间错乱。用自然语言表达效果最好，专业摄影术语反而让模型理解变差。**

| 提取项 | 正确写法示例 | 错误写法 |
| --- | --- | --- |
| 路/步道动线 | "路是从左往右横着走的" 或 "路从左下方往右上方斜着延伸" | "道路在右边" |
| 护栏/绿植等 | "蓝白护栏和绿色灌木丛在路的上面，就在路旁边" | "护栏在右侧" |
| 电动车/道具 | "电动车停在路边靠近镜头这侧，偏左边" | "电动车在左边" |
| 角色入画方向 | "角色从左边走进来，顺着路往右边走" | "角色从左边进来"（缺移动方向） |
| 背景建筑 | "住宅楼在远处后面，在画面上方" | "楼在后面" |

**完整开场描述示例**：`路从左下方往右上方斜着延伸，护栏和绿化灌木在路的上面紧靠着路边，电动车停在路边靠近镜头这侧偏左。远处后面是住宅楼。角色从左边走进来，顺着路往右边走。`

**季节与服装适配**（出提示词时必做）：

对照当前真实日期（`今日日期` 字段），判断季节，替换原视频中不符合当季的服装和场景描述：

| 季节 | 服装方向 | 场景/植物 |
| --- | --- | --- |
| 夏季（6-8月） | 短袖/轻薄上衣/连衣裙，不写毛衣/开衫/羽绒 | 绿植茂盛，可写晴天/阵雨，不写落叶/积雪 |
| 秋季（9-11月） | 薄外套/长袖，不写厚羽绒 | 可写落叶，不写积雪/茂盛绿植 |
| 冬季（12-2月） | 厚外套/羽绒，不写短袖 | 可写阴天/小雪，不写鲜花盛开 |
| 春季（3-5月） | 薄外套/轻薄长袖，不写厚羽绒 | 可写花开/新叶，不写积雪/枯黄 |

原视频服装/场景与当季不符时，**改写描述，不说明原视频是什么**，直接给符合当季的版本。

**时长预检**（写提示词前先算，超出就不写）：

- 台词秒数估算：台词字数 ÷ 4.5（普通话正常语速）
- 动作秒数：从分析报告第4段取各动作时长相加
- 台词秒数 + 动作秒数 ≤ 10s → 正常写豆包提示词
- 超出10s → 提示用户"台词+动作约Xs，略超豆包上限，建议①裁剪台词至约X字以内，或②改用可灵"，**用户确认继续则直接写，不再追问**

**结构顺序**：时长预检 → 技术头 → **开场构图** → 场景 → 角色+参考图说明 → 动作序列 → 台词（嵌入动作中） → 书动作 → 收尾约束

### 可灵版（>10s，逐镜输出）

**触发条件**：视频总时长 >10s，或用户明确指定。

**@引用固定集合**（每镜都带）：

- 第一镜：`@构图参考图`（用户提供的景别/占位参考，通常是原视频第一帧替换后的图）
- 后续镜：`@上一镜最后帧`（替代上一镜的构图参考图，保证拼接流畅）
- 每镜：`@角色图`、`@场景图`、`@道具书图`

**每镜结构顺序**：

1. 技术头
2. 起始画面（第二镜起写：`以上一镜最后帧为起始画面`）
3. 场景 + 角色动作 @引用
4. 台词（当前镜所属段落）
5. 书：动作 + 空间姿态 + 封面朝向 + 禁止翻开 @道具书图
6. 运镜 + 时长估算（约Xs）
